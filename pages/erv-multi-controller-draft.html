Multi-Controller for a Single Broan ERV

Smart sequencing — the VTouch bridge as a reverse-engineering tool first. Capture the full register map before writing controller firmware, then graduate the same hardware into a permanent proxy.

Updated System — 3 Panels + Web UI

The controller node is the only RS-485 master. All other devices are MQTT clients. The controller validates, queues, and serializes commands, then publishes authoritative state and a full audit log so every client stays synchronized.

High-Level Architecture

[Client A: VTTOUCH Panel]      [Client B: ESP32-S3 Buttons]     [Client C: Web UI]
              │                            │                             │
              └─────────────── MQTT (commands + state + audit log) ───────┘
                                             │
                                   [Controller: ESP32-S3 or Pi]
                                             │
                                      RS-485 to ERV

MQTT Topic Schema

Topic Direction Retained Description
erv/command/request Client → Controller No Command payload from any client
erv/command/ack/{client_id} Controller → Client No Per-client ACK with status
erv/state Controller → All Yes Authoritative ERV state snapshot
erv/audit Controller → All No Append-only event log
erv/errors Controller → All No Bus faults, rejected commands
erv/config/clients/{client_id} Admin → Controller Yes Per-client config (read-only flag, priority)
erv/controller/status Controller → All Yes Controller heartbeat + uptime

Retained erv/state lets clients sync instantly on reconnect.

Conflict Resolution

  1. Priority wins: lower number beats higher number.
  2. Recency wins: if priority matches, later timestamp wins.
  3. Safety wins: invalid commands are rejected outright.

Superseded commands receive an ACK with status = superseded.

Command Queue

  • FIFO queue, suggested depth: 5
  • Per-entry TTL: 10 seconds
  • Expired commands are dropped with expired ACK
  • One RS-485 transaction at a time

Webserver Layer

REST Endpoints

  • GET /api/state — current erv/state
  • POST /api/command — same schema as MQTT command request
  • GET /api/audit?limit=50 — recent audit from SQLite
  • GET /api/config — client configuration snapshot

WebSocket

  • /ws/state — push on every poll or command execution
  • Browser UI stays live without polling
  • Host UI directly from controller storage

Dedicated ESP32-S3 Remote

Suggested BOM

  • ESP32-S3 dev board (FeatherS3 or XIAO ESP32-S3)
  • 0.96\" or 1.3\" OLED (SSD1306/SH1106)
  • 4–5 tactile buttons
  • Optional WS2812B status LED
  • USB-C power or LiPo + TP4056

Firmware Behavior

  • Boot: Wi-Fi → MQTT → subscribe to state + ACK
  • Button press: debounce, publish command, await ACK
  • Offline mode: show banner, queue up to 3 commands
  • Display: mode + speed, temps + humidity, last source

Open Questions & Decisions

  • VTTOUCH control mode: Default to command-capable, allow read-only via erv/config/clients/vttouch.
  • Conflict handling: Reject unsafe commands with explicit reasons.
  • Audit logging: MQTT for live stream + SQLite for durable history.
  • Poll interval: Suggested 5–10 seconds; confirm ERV firmware tolerance.
  • Broker location: On controller (standalone) vs LAN broker (shared infra).
  • OTA updates: Controller + remotes should support OTA.

Phase 1: VTouch Bridge as Clean-Room Reverse Engineering Tool

Build this before anything else. It serves two purposes: protocol capture while the VTouch is still the ERV master, and a permanent VTouch proxy once the register map is validated.

What You're Capturing

  • Baud rate + framing
  • Modbus function codes
  • Register addresses tied to control functions
  • Timing: poll interval, response time, inter-frame gaps

Hardware Wiring

[VTouch] ──A/B/GND──> [ESP32-S3 #2, UART0] ──A/B/GND──> [MAX3485, UART1] ──A/B/GND──> [ERV]

UART0 faces VTouch, UART1 faces ERV. Capture mode is listen-only on both sides.

Parts for Phase 1 Build

Part Notes Cost

2× MAX3485 or SP3485 breakout One per RS-485 side — 3.3V compatible $5–8
Small protoboard or PCB Perfboard is fine for RE work $2–4
USB-C for power + serial debug Can power from VTouch 12V line via buck converter —

Total: ~$15–22

Wire the DE/RE pins on each transceiver to separate GPIO so you can control direction independently. UART0 faces VTouch (listen-only in Phase 1 — tie DE low, RE low). UART1 faces ERV (also listen-only initially — you are just observing, not injecting anything yet).

Phase 1 Firmware: Passive Capture Mode

// Both UARTs in receive-only mode
Serial1.begin(9600, SERIAL_8N1, RX0_PIN, TX0_PIN); // VTouch side
Serial2.begin(9600, SERIAL_8N1, RX1_PIN, TX1_PIN); // ERV side

struct Frame {
  uint32_t timestamp_ms;
  char     source;   // 'V' = VTouch, 'E' = ERV response
  uint8_t  data[32];
  uint8_t  len;
};

// Main loop: drain both UARTs, buffer into frames
// Frame boundary = inter-frame gap > 3.5 char times (~4ms at 9600)
// Publish each complete frame to serial and/or erv/capture MQTT topic

void publishFrame(Frame& f) {
  // JSON: {"ts":12345,"src":"V","hex":"02 03 00 01 00 06 xx xx"}
  // Publish to erv/raw/vtx or erv/raw/erv
}

The 3.5 character gap rule is standard Modbus RTU frame detection. At 9600 baud that is ~4ms of silence.

The Capture Session

Connect the bridge between VTouch and ERV. Open MQTT Explorer and watch erv/raw/#. Then operate the VTouch through every function you care about.

Session checklist — do each action several times:

  • Idle for 2 minutes (poll cycle + heartbeat)
  • Mode: Off → Low → Med → High → Off
  • Trigger Boost, let run, cancel manually
  • Observe supply/exhaust temperature views
  • Observe humidity readout
  • Check filter status indicator
  • Power cycle VTouch and capture init sequence

Decoding the Capture

[Addr][FuncCode][RegHi][RegLo][CountHi][CountLo][CRC_Lo][CRC_Hi]
  01     03       00     01     00        06       XX       XX

ERV response to a read:

[Addr][FuncCode][ByteCount][Data...][CRC_Lo][CRC_Hi]
  01     03        0C        [12 bytes of data]  XX XX

Write operations (0x06) map to control registers. Read responses that change with sensor values map to sensor registers.

Publish Decoded Frames (Parallel)

{"ts":12340,"src":"V","fn":"read","reg":"0x0003","count":6}
{"ts":12345,"src":"E","fn":"read_resp","reg":"0x0003",
 "values":{"0x0003":682,"0x0004":712,"0x0005":52}}

Values like 682 and 712 likely represent °F × 10.

  0x0002  R   Fan speed actual (RPM or index TBD)
  0x0003  R   Supply air temp  (°F × 10)
  0x0004  R   Exhaust air temp (°F × 10)
  0x0005  R   Relative humidity (%)
  0x0006  R   Filter status    (0=ok 1=replace)
  0x0007  RW  Boost timer      (minutes remaining)
  0x0008  R   Defrost active   (0/1)
  [...]    —  Unknown — log raw value, label TBD

ERV response time: <30ms (field measured)
Max inter-frame gap tolerance: TBD

Mark every register as field-verified with a date. Inferred values should be tagged.

Phase 1.5: Active Proxy

VTouch READ  → Bridge forwards to ERV → ERV responds → Bridge forwards to VTouch
VTouch WRITE → Bridge:
  1. Forwards to ERV (executes immediately)
  2. Publishes to erv/command/request with decoded payload
  3. Forwards ERV's WRITE ACK back to VTouch

Phase 2: Full Proxy (Controller Takes Over)

VTouch READ  → Bridge answers from local erv/state cache (not forwarded to real ERV)
VTouch WRITE → Bridge publishes to erv/command/request → Controller executes
Controller publishes erv/state → Bridge updates cache → satisfies VTouch's next READ

Web UI Hosted on ESP32-S3 #1

Stack: ESPAsyncWebServer (HTTP + WebSocket) + LittleFS (static UI). WebSocket pushes state updates with no polling.

UI Surface

  • Live status panel: mode, fan speed, temps, humidity, filter status
  • Control panel: mode buttons + boost duration
  • Audit log: last 50 commands with outcome
  • Client status: online panels, last seen, broker count
  • Admin panel: poll interval, read-only toggles, export CSV
