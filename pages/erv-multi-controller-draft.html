Custom Controller for
Broan B210E75RT AI ERV

Full clean-room reverse engineering of the VTouch RS-485 protocol, followed by a staged replacement with an ESP32-S3 RS-485 master running an MQTT broker, a web dashboard, and drop-in VTouch-emulator panels — all without the ERV knowing anything changed.

Broan B210E75RT ESP32-S3 RS-485 · Modbus RTU MQTT WT32-SC01 Plus

What You End Up With

Controller Node

ESP32-S3 as the only RS-485 master. Talks directly to the ERV, hosts MQTT broker + web UI.

VTouch Emulator Bridge

Second ESP32-S3 intercepts the original VTouch, translates its commands to MQTT, and feeds it fabricated responses so it still works normally.

Replacement Panels

WT32-SC01 Plus touchscreens that look and feel like the VTouch but run your firmware. Multiple panels possible.

Web Dashboard

Hosted on the controller at http://erv.local. Live state, controls, audit log, config — from any browser on the LAN.

Only One RS-485 Master

ESP32-S3 #1 is the sole Modbus master. Nothing else writes to the ERV bus directly.

VTouch Sees No Change

The bridge responds to the VTouch's Modbus reads/writes from a cached state mirror. VTouch thinks it's still the master.

All Clients in Sync

Controller publishes authoritative state to erv/state (retained). Every client — VTouch bridge, panels, browser — stays synchronized.

Stage 0 — Development Environment Setup

Est. time: 1–2 hours · Do this before touching the ERV.

0.1 — Install Toolchain

  1. Install VS Code + PlatformIO extension. This is the recommended toolchain for this build — better library management than Arduino IDE and supports both boards.
  2. Or use Arduino IDE 2.x with the ESP32 board package by Espressif (version 2.x or 3.x) installed via Boards Manager. URL: https://raw.githubusercontent.com/espressif/arduino-esp32/gh-pages/package_esp32_index.json
  3. Install required libraries (PlatformIO: add to platformio.ini · Arduino IDE: Library Manager):
    • knolleary/PubSubClient — MQTT client
    • me-no-dev/ESPAsyncWebServer + me-no-dev/AsyncTCP — web server + WebSocket
    • olehs/UMODBUS or eModbus/eModbus — Modbus RTU master (Stage 2+)
    • ArduinoJson — JSON serialization
    • ESPmDNS — built-in with ESP32 package
  4. For Stage 4 (WT32-SC01 Plus panels): install LVGL 8.x and the WT32-SC01 Plus board definitions.

0.2 — Flash Test Both Boards

Before touching the ERV, verify both ESP32-S3 boards enumerate over USB and accept a basic blink sketch. Identify serial port assignments on your machine.

Board: "XIAO_ESP32S3"  (for capture bridge board)
Board: "Adafruit Feather ESP32-S3"  (for controller board)
Upload speed: 921600
USB CDC on Boot: Enabled  ← important for Serial monitor on S3

0.3 — Wire the Capture Bridge (Bench Test First)

Build and test the bridge on the bench before connecting it to the ERV. Use two 3.3 V MAX3485 breakouts and the XIAO ESP32-S3.

XIAO ESP32-S3 GPIO pins used:
  GPIO4  = UART1_RX  (from MAX3485 #1 RO — VTouch side)
  GPIO5  = UART1_TX  (to MAX3485 #1 DI  — VTouch side)
  GPIO6  = DIR1      (DE/RE control for MAX3485 #1)

  GPIO7  = UART2_RX  (from MAX3485 #2 RO — ERV side)
  GPIO8  = UART2_TX  (to MAX3485 #2 DI  — ERV side)
  GPIO9  = DIR2      (DE/RE control for MAX3485 #2)

MAX3485 wiring (each transceiver):
  VCC → 3.3 V
  GND → GND
  A   → RS-485 A terminal
  B   → RS-485 B terminal
  RO  → ESP32-S3 RX pin
  DI  → ESP32-S3 TX pin
  DE+RE tied together → DIR GPIO  (tie low for receive-only in Stage 1)
Stage 1 is listen-only. Tie DE low and RE low on BOTH transceivers during capture phase. The bridge cannot transmit — it can only observe. This is the "clean room" constraint that prevents accidental ERV commands during RE.

Stage 1 — Passive Capture & Protocol Mapping

Est. time: 2–4 hours · VTouch remains in circuit. Bridge is read-only.

1.1 — Physical Install of the Bridge

  1. Power down the ERV at the breaker.
  2. Open the ERV service panel. Locate the VTouch terminal block (A, B, GND, +12 V).
  3. Use T-tap wire connectors or a 4-way terminal splitter to insert the bridge in series between the ERV terminal and the VTouch cable — do not cut the VTouch cable.
  4. Connect bridge wiring: 
    ERV terminal ─── [T-tap A] ──┬── VTouch cable A
                              └── MAX3485 #2 (ERV side) A

ERV terminal ─── [T-tap B] ──┬── VTouch cable B
                              └── MAX3485 #2 (ERV side) B

VTouch cable A ──────────────── MAX3485 #1 (VTouch side) A
VTouch cable B ──────────────── MAX3485 #1 (VTouch side) B
ERV GND ─────────────────────── Bridge GND
ERV +12 V ──── Buck 12→5 V ──── Bridge USB-C / 5 V rail
  5. Power up the ERV. VTouch should function normally. The bridge is invisible — it only listens.

1.2 — Stage 1 Firmware: Passive Capture

// platformio.ini board: seeed_xiao_esp32s3
// Both UARTs receive-only. DE=LOW, RE=LOW on both MAX3485 boards.

#include <WiFi.h>
#include <PubSubClient.h>

#define UART_BAUD   9600
#define RX_VTOUCH   4     // UART1 RX (VTouch side)
#define TX_VTOUCH   5     // UART1 TX (unused in Stage 1)
#define DIR_VTOUCH  6     // DE/RE — keep LOW
#define RX_ERV      7     // UART2 RX (ERV side)
#define TX_ERV      8     // UART2 TX (unused in Stage 1)
#define DIR_ERV     9     // DE/RE — keep LOW

HardwareSerial vtSerial(1);   // UART1 — VTouch side
HardwareSerial ervSerial(2);  // UART2 — ERV side

const char* WIFI_SSID = "YOUR_SSID";
const char* WIFI_PASS = "YOUR_PASSWORD";
const char* MQTT_HOST = "192.168.1.XXX"; // controller ESP32 IP (temp broker)

WiFiClient   net;
PubSubClient mqtt(net);

uint8_t vtBuf[64], ervBuf[64];
uint8_t vtLen = 0, ervLen = 0;
uint32_t vtLast = 0, ervLast = 0;
const uint32_t FRAME_GAP_MS = 5; // 3.5 char times @ 9600 baud ≈ 4 ms

void setup() {
  Serial.begin(115200);
  pinMode(DIR_VTOUCH, OUTPUT); digitalWrite(DIR_VTOUCH, LOW);
  pinMode(DIR_ERV,    OUTPUT); digitalWrite(DIR_ERV, LOW);
  vtSerial.begin(UART_BAUD, SERIAL_8N1, RX_VTOUCH, TX_VTOUCH);
  ervSerial.begin(UART_BAUD, SERIAL_8N1, RX_ERV, TX_ERV);
  WiFi.begin(WIFI_SSID, WIFI_PASS);
  while (WiFi.status() != WL_CONNECTED) delay(200);
  mqtt.setServer(MQTT_HOST, 1883);
  mqtt.connect("erv-bridge");
}

void publishFrame(const char* topic, uint8_t* buf, uint8_t len, uint32_t ts) {
  char json[200];
  char hex[100] = {};
  for (int i = 0; i < len; i++) sprintf(hex + i*3, "%02X ", buf[i]);
  snprintf(json, sizeof(json),
    "{\"ts\":%lu,\"hex\":\"%s\",\"len\":%d}", ts, hex, len);
  mqtt.publish(topic, json);
  Serial.println(json); // also to serial monitor
}

void drainUart(HardwareSerial& s, uint8_t* buf, uint8_t& len,
               uint32_t& lastMs, const char* topic) {
  while (s.available()) {
    buf[len++] = s.read();
    lastMs = millis();
  }
  if (len > 0 && (millis() - lastMs) > FRAME_GAP_MS) {
    publishFrame(topic, buf, len, lastMs);
    len = 0;
  }
}

void loop() {
  if (!mqtt.connected()) mqtt.connect("erv-bridge");
  mqtt.loop();
  drainUart(vtSerial,  vtBuf,  vtLen,  vtLast,  "erv/raw/vt");
  drainUart(ervSerial, ervBuf, ervLen, ervLast, "erv/raw/erv");
}

Flash this, open MQTT Explorer pointed at the broker, then operate the VTouch through every function.

1.4 — Decoding Modbus RTU Frames

All frames follow standard Modbus RTU framing. Decode each captured hex string:

── Read Request (VTouch → ERV) ──
[Addr][FC  ][RegHi][RegLo][CntHi][CntLo][CRC_Lo][CRC_Hi]
  01    03    00     01     00     06      XX       XX
FC 03 = Read Holding Registers
Reg 0x0001, count 6 → reading 6 registers starting at 0x0001

── Read Response (ERV → VTouch) ──
[Addr][FC  ][ByteCnt][Data × N bytes ][CRC_Lo][CRC_Hi]
  01    03     0C     [12 bytes / 6 regs]  XX     XX

── Write Request (VTouch → ERV) ──
[Addr][FC  ][RegHi][RegLo][ValHi][ValLo][CRC_Lo][CRC_Hi]
  01    06    00     01     00     03      XX       XX
FC 06 = Write Single Register
Reg 0x0001, value 0x0003 → set mode to High (or index 3)

── Write Response (ERV echoes request) ──
  01    06    00     01     00     03      XX       XX
Values like 0x02AC (684) on a temperature register likely mean 68.4 °F (°F × 10). Values like 0x0034 (52) on a humidity register are direct percentage. Document each register as you confirm it.

2.3 — Upgrade Bridge to Active Proxy

In Stage 2 the bridge still passes VTouch commands through to the ERV, but it also translates them to MQTT so the controller is aware.

VTouch READ  → Bridge forwards bytes verbatim to ERV side
             ← ERV response forwarded back to VTouch
             → Decoded frame published to erv/audit (info only)

VTouch WRITE → Bridge:
  1. Forwards to ERV (executes on real unit)
  2. Publishes decoded command to erv/command/request
  3. Forwards ERV's WRITE echo back to VTouch
This is the validation phase. Watch the controller's MQTT subscription to erv/command/request and confirm it is decoding VTouch write frames correctly before proceeding to Stage 3.

Stage 3 — Controller Takeover & VTouch Emulation

Est. time: 2–4 hours · Controller becomes sole RS-485 master. Bridge feeds VTouch fabricated responses.

3.1 — Rewire the Bridge

Remove the bridge's tap on the ERV side. The controller is now the only device on the ERV RS-485 bus. The bridge only talks to the VTouch.

Before Stage 3:
  VTouch ── Bridge (pass-through) ── ERV

After Stage 3:
  VTouch ── Bridge (VTouch emulator)
                                      Controller ── ERV

The bridge now has a single MAX3485 (VTouch side only). It no longer has physical access to the ERV bus.

3.2 — Bridge Firmware: VTouch Emulator

The bridge now acts as a fake ERV slave to the VTouch. It answers all of VTouch's Modbus reads from a local state cache that is kept in sync with erv/state via MQTT. VTouch writes are translated to MQTT commands and sent to the real controller.

// Bridge ESP32-S3 #2 — Stage 3: VTouch emulator
// Only one UART now (VTouch side). DE/RE controlled for half-duplex.

// ── State cache (mirror of erv/state) ──────────
struct ERVState {
  uint8_t  mode;          // 0–4
  uint16_t supply_temp;   // °F × 10
  uint16_t exhaust_temp;  // °F × 10
  uint8_t  humidity;      // %
  uint8_t  filter_status; // 0=ok 1=replace
  uint16_t boost_remaining;
  uint8_t  defrost;
} cache;

// ── MQTT: subscribe to erv/state, update cache ──
void onMqttState(const char* payload) {
  // Parse JSON → update cache fields
}

// ── Modbus slave responder ────────────────────
// Listen on VTouch serial for requests
// When VTouch sends FC03 (read), reply with cache values
// When VTouch sends FC06 (write), publish to erv/command/request
//   then reply with Modbus write echo (confirm to VTouch immediately)

void handleVTouchRequest(uint8_t* req, uint8_t len) {
  uint8_t fc  = req[1];
  uint16_t reg = (req[2] << 8) | req[3];

  if (fc == 0x03) {          // READ — answer from cache
    uint8_t resp[64];
    uint8_t cnt = buildReadResponse(reg, req[4]<<8|req[5], cache, resp);
    sendRS485(resp, cnt);
  } else if (fc == 0x06) {   // WRITE — forward via MQTT
    uint16_t val = (req[4] << 8) | req[5];
    publishCommand(reg, val);
    // Echo write confirmation back to VTouch immediately
    uint8_t echo[8];
    memcpy(echo, req, 6);
    appendCRC(echo, 6);
    sendRS485(echo, 8);
  }
}
The VTouch poll interval is ~5 seconds. The controller also polls the ERV every 5 seconds and publishes to erv/state. The bridge's cache will be at most 5–10 seconds stale — completely imperceptible to the VTouch.

3.3 — Command Arbitration on the Controller

All commands — from VTouch bridge, replacement panels, and web UI — arrive on erv/command/request. The controller serializes them before issuing RS-485 writes.

Arbitration Rules

  1. Lower priority number wins (0 = highest)
  2. Same priority → later timestamp wins
  3. Invalid register/value → reject outright
  4. Superseded commands get status: superseded ACK

Queue Params

  • FIFO depth: 5 entries
  • Per-entry TTL: 10 seconds
  • One RS-485 transaction at a time
  • Expired entries → status: expired ACK

3.4 — Smoke Test Checklist

  • ☐ VTouch shows correct mode, temps, humidity (sourced from bridge cache)
  • ☐ Pressing a mode button on VTouch → command appears in MQTT → ERV changes mode → controller publishes new state → bridge cache updates → VTouch confirms on next poll
  • ☐ Browser at http://erv.local shows matching state in real time
  • ☐ Command sent from web UI changes ERV mode → VTouch reflects update within one poll cycle
  • ☐ Boost from VTouch, cancelled from web UI — both sides reflect correctly
  • ☐ Power cycle controller — VTouch remains functional (bridge queues reads, reconnects to broker)

  • LVGL ready: Works with LVGL 8.x out of the box, extensive community examples
  • Well supported: Arduino + ESP-IDF, active GitHub repo from Wireless-Tag
  • Right size: 3.5" gives a genuinely usable dashboard — 320×240 (2.8") can feel cramped
  • Mountable: 3D-printed Decora faceplate files exist for it; fits standard single-gang
  • Powered by USB-C: route a short cable through the wall cavity from a nearby outlet or in-wall USB adapter

4.2 — Physical Installation

  1. Cut a single-gang opening in drywall at the desired location (where VTouch was, or new location).
  2. Install an old-work single-gang electrical box. Run USB-C power cable through a low-voltage opening or the knockout.
  3. If replacing the original VTouch location: the existing VTouch cable (A/B/GND/12V) can either be capped (if bridge emulator handles the VTouch) or repurposed for panel power if convenient.
  4. 3D print the Decora-size faceplate for the WT32-SC01 Plus. Secure the board to the faceplate with M2 standoffs (or hot-glue for prototype).
  5. Connect USB-C, fold into the box, snap the faceplate to the Decora plate, and screw to the single-gang box.
The WT32-SC01 Plus draws about 300–400 mA at 5 V when the display is on full brightness. A 1 A in-wall USB adapter is sufficient. Dim the display to 30–40% in the firmware to reduce power and prevent image retention.

4.5 — Multi-Panel Setup

Each additional WT32-SC01 Plus is an independent MQTT client. Add one per location — upstairs hall, kitchen, wherever you want VTouch-like control. All panels stay in sync via erv/state retained topic.

  • Each panel has its own client_id (e.g. panel-kitchen, panel-hallway)
  • Use captive-portal first-boot config to set Wi-Fi credentials without reflashing
  • OTA update all panels from a single binary via ArduinoOTA or custom HTTP endpoint
  • If you keep the original VTouch: its commands arrive via the bridge with client_id: vttouch-bridge and participate in normal arbitration — same priority as any other client

MQTT Topic Schema

TopicDirectionRetainedDescription
erv/command/requestAny client → ControllerNoCommand payload from any client (panel, web, bridge)
erv/command/ack/{client_id}Controller → ClientNoPer-client ACK: accepted / rejected / superseded / expired
erv/stateController → AllYesAuthoritative ERV state snapshot — clients sync on reconnect
erv/auditController → AllNoAppend-only event log (command + outcome + RS-485 raw)
erv/errorsController → AllNoBus faults, rejected commands, Modbus errors
erv/raw/vtBridge → ControllerNoStage 1 only — raw hex frames from VTouch (RE capture)
erv/raw/ervBridge → ControllerNoStage 1 only — raw hex frames from ERV (RE capture)
erv/config/clients/{client_id}Admin → ControllerYesPer-client config: read-only flag, priority override
erv/controller/statusController → AllYesHeartbeat + uptime + broker client count

  • GET /api/audit?limit=50 — recent log

OTA Updates & First-Boot Configuration

First-Boot Captive Portal (Panels)

So you never need to reflash a wall panel just to change Wi-Fi credentials. Use WiFiManager:

  1. On first boot (or if saved credentials fail), panel broadcasts AP: ERV-Panel-Setup
  2. Connect from phone → captive portal opens
  3. Enter Wi-Fi SSID, password, controller IP, client name
  4. Panel saves to NVS, reboots, connects to broker

OTA via ArduinoOTA

All ESP32-S3 nodes (controller + bridge + panels) should include OTA support so you never need physical USB access after initial flash.

#include <ArduinoOTA.h>
// In setup():
ArduinoOTA.setHostname("erv-controller");
ArduinoOTA.setPassword("your-ota-pass");
ArduinoOTA.begin();
// In loop():
ArduinoOTA.handle();

Use PlatformIO upload_protocol = espota for seamless OTA deploys.

  1. 01 Order Stage 0–1 parts (XIAO ESP32-S3 + 2× MAX3485 breakouts)
  2. 02 Set up toolchain, flash test boards, verify USB enumeration
  3. 03 Wire capture bridge (two MAX3485 transceivers, DE/RE tied low)
  4. 04 Flash Stage 1 firmware — passive capture mode
  5. 05 Run full VTouch capture session, export MQTT log
  6. 06 Decode frames, fill in register map, confirm baud + slave addr
  7. 07 Order Stage 2–3 parts (Feather ESP32-S3 controller + single MAX3485)
  8. 08 Wire and flash controller — MQTT broker + Modbus RTU master
  9. 09 Upgrade bridge to active-proxy mode (Stage 2), validate command flow
  10. 10 Rewire bridge as VTouch emulator (Stage 3) — controller takes over RS-485
  11. 11 Run Stage 3 smoke test checklist — all clients in sync
  12. 12 Order WT32-SC01 Plus panel(s) + mounting hardware

  15. 15 Mount panel(s), verify all clients sync. Optionally decommission original VTouch.

Open Questions to Resolve in Stage 1

  • Confirmed baud rate: 9600 is assumed — verify from capture. Some Broan units use 19200.
  • Slave address: 0x01 is the Modbus default — verify from first byte of every VTouch request.
  • VTouch poll interval: Measure inter-request gap from MQTT timestamps. Expected ~5 s.
  • All writable registers: Mode (0x0001) and boost timer (0x0007) are expected. Are there registers for fan speed independent of mode? Verify.
  • Defrost behavior: Does the ERV reject writes during defrost cycle? Capture if a defrost runs during the session.
  • VTouch init sequence: Does it send any special frame on power-up that the emulator must replicate? Capture a VTouch power cycle.
  • Broker memory: Test uMQTTBroker under load (3 clients + WebSocket + REST) on Feather ESP32-S3. Upgrade to Pi Zero 2 W as broker if RAM is tight.
