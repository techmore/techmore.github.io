Multi-Controller for a Single Broan ERV

A hub-and-spoke MQTT architecture that enables multiple control surfaces to command a single Broan ERV without violating the RS-485 single-master constraint.

Overview

The controller node is the only RS-485 master. All other devices are MQTT clients. The controller validates, queues, and serializes commands, then publishes authoritative state and audit logs so every client stays synchronized.

Roles & Responsibilities

  • Controller Node (ESP32-S3 or Pi Zero 2W): Sole RS-485 master. Polls ERV, validates/queues commands, publishes state, logs audit trail.
  • Hardware Remote Client (ESP32-S3): Buttons + OLED/TFT, publishes commands, mirrors state.
  • VTTOUCH Panel Client: Command-capable by default, can be flipped to read-only via retained MQTT config.
  • Web UI / Webserver: REST + WebSocket surface for browser UI and automation clients.

Why This Works

  • Single RS-485 master preserves ERV firmware constraints.
  • MQTT gives shared state, command logging, and multi-client sync.
  • Command arbitration prevents conflicts while keeping UI responsive.

High-Level Architecture

[Client A: VTTOUCH Panel]      [Client B: ESP32-S3 Buttons]     [Client C: Web UI]
              │                            │                             │
              └─────────────── MQTT (commands + state + audit log) ───────┘
                                             │
                                   [Controller: ESP32-S3 or Pi]
                                             │
                                      RS-485 to ERV

MQTT Topic Schema

Topic Direction Retained Description
erv/command/request Client → Controller No Command payload from any client
erv/command/ack/{client_id} Controller → Client No Per-client ACK with status
erv/state Controller → All Yes Authoritative ERV state snapshot
erv/audit Controller → All No Append-only event log
erv/errors Controller → All No Bus faults, rejected commands
erv/config/clients/{client_id} Admin → Controller Yes Per-client config (read-only flag, priority)
erv/controller/status Controller → All Yes Controller heartbeat + uptime

Retained erv/state lets new clients instantly sync without waiting for the next poll.

Payload Schemas

Command Request

erv/command/request

{
  "client_id": "esp32-remote-01",
  "request_id": "a3f9c1",
  "timestamp": 1749500000,
  "command": "set_mode",
  "params": {
    "mode": "boost",
    "duration_minutes": 20
  },
  "priority": 1
}

Command ACK

erv/command/ack/{client_id}

{
  "request_id": "a3f9c1",
  "status": "accepted",
  "queued_position": 1,
  "reason": null
}

Status values: accepted, rejected, superseded, expired.

ERV State

erv/state

{
  "timestamp": 1749500005,
  "mode": "boost",
  "fan_speed": 3,
  "supply_temp_f": 68,
  "exhaust_temp_f": 71,
  "humidity_pct": 52,
  "boost_remaining_min": 19,
  "filter_status": "ok",
  "bus_ok": true,
  "last_command_source": "esp32-remote-01"
}

Audit Entry

erv/audit

{
  "timestamp": 1749500000,
  "client_id": "esp32-remote-01",
  "request_id": "a3f9c1",
  "command": "set_mode",
  "params": { "mode": "boost", "duration_minutes": 20 },
  "outcome": "executed",
  "rs485_raw": "0x02 0x06 0x00 0x01 0x00 0x03"
}

Conflict Resolution

  1. Priority wins: lower number beats higher number.
  2. Recency wins: if priority matches, later timestamp wins.
  3. Safety wins: invalid or unsafe commands are rejected outright.

Superseded commands receive an ACK with status = superseded.

Command Queue

  • FIFO queue, suggested depth: 5
  • Per-entry TTL: 10 seconds
  • Commands expire if not executed in time
  • One RS-485 transaction at a time

Webserver Layer

REST Endpoints

  • GET /api/state — current erv/state
  • POST /api/command — same schema as MQTT command request
  • GET /api/audit?limit=50 — recent audit entries from SQLite
  • GET /api/config — client configuration snapshot

WebSocket

  • /ws/state — pushes state updates on every poll or command execution
  • Browser UI stays live without polling
  • Serve UI directly from controller storage for a local-only stack

Dedicated ESP32-S3 Remote

Suggested BOM

  • ESP32-S3 dev board (FeatherS3 or XIAO ESP32-S3)
  • 0.96" or 1.3" OLED (SSD1306/SH1106)
  • 4–5 tactile buttons
  • Optional WS2812B status LED
  • USB-C power or LiPo + TP4056

Firmware Behavior

  • Boot: Wi-Fi → MQTT → subscribe to state + ACK
  • Button press: debounce, publish command, await ACK
  • Offline mode: show banner, queue up to 3 commands
  • Display: mode + speed, temps + humidity, last command source

Open Questions & Decisions

  • VTTOUCH control mode: Default to command-capable, allow read-only via erv/config/clients/vttouch.
  • Conflict handling: Reject unsafe commands with explicit reasons (avoid silent coercion).
  • Audit logging: MQTT for live stream + SQLite for durable history.
  • Poll interval: Suggested 5–10 seconds; confirm ERV firmware tolerances.
  • Broker location: On controller (standalone) vs LAN broker (shared infra).
  • OTA: Controller + remotes should support OTA for maintenance.
