Multi-Controller for a Single Broan ERV

A hub-and-spoke MQTT architecture that enables multiple control surfaces to command a single Broan ERV while keeping a strict single RS-485 master.

Updated System — 3 Panels + Web UI

The controller node is the only RS-485 master. All other devices are MQTT clients. The controller validates, queues, and serializes commands, then publishes authoritative state and an audit log so every client stays synchronized.

High-Level Architecture

[Client A: VTTOUCH Panel]      [Client B: ESP32-S3 Buttons]     [Client C: Web UI]
              │                            │                             │
              └─────────────── MQTT (commands + state + audit log) ───────┘
                                             │
                                   [Controller: ESP32-S3 or Pi]
                                             │
                                      RS-485 to ERV

Phase 1: VTouch Bridge as Clean-Room Reverse Engineering Tool

Build this before anything else. It serves as a protocol capture device while the VTouch is still the ERV master, and later becomes the permanent VTouch proxy. Same hardware, firmware evolves in place.

What You're Capturing

  • Baud rate + framing
  • Modbus function codes
  • Register addresses tied to each control function
  • Timing: poll interval, response time, inter-frame gaps

Hardware Wiring

[VTouch] ──A/B/GND──> [ESP32-S3 #2, UART0]
                     └─A/B/GND──> [MAX3485, UART1] ──A/B/GND──> [ERV]

Two RS-485 transceivers on one ESP32-S3. UART0 faces VTouch, UART1 faces ERV.

Phase 1 Firmware: Passive Capture Mode

// Both UARTs in receive-only mode
Serial1.begin(9600, SERIAL_8N1, RX0_PIN, TX0_PIN); // VTouch side
Serial2.begin(9600, SERIAL_8N1, RX1_PIN, TX1_PIN); // ERV side

struct Frame {
  uint32_t timestamp_ms;
  char     source;   // 'V' = VTouch, 'E' = ERV response
  uint8_t  data[32];
  uint8_t  len;
};

// Main loop: drain both UARTs, buffer into frames
// Frame boundary = inter-frame gap > 3.5 char times (~4ms at 9600)
// Publish each complete frame to serial and/or erv/capture MQTT topic

void publishFrame(Frame& f) {
  // JSON: {"ts":12345,"src":"V","hex":"02 03 00 01 00 06 xx xx"}
  // Publish to erv/raw/vtx or erv/raw/erv
}

3.5 character gap rule is standard Modbus RTU frame detection.

Capture Session Checklist

  • Idle for 2 minutes (captures poll cycle)
  • Mode: Off → Low → Med → High → Off
  • Trigger Boost, let run, cancel manually
  • Observe supply/exhaust temperature views
  • Observe humidity readout
  • Check filter status indicator
  • Power cycle VTouch and capture init sequence

Decoding the Capture

[Addr][FuncCode][RegHi][RegLo][CountHi][CountLo][CRC_Lo][CRC_Hi]
  01     03       00     01     00        06       XX       XX

ERV response to read:

[Addr][FuncCode][ByteCount][Data...][CRC_Lo][CRC_Hi]
  01     03        0C        [12 bytes of data]  XX XX

Build a register map as you decode. Write commands (0x06) map to control registers. Changing sensor values map to read registers.

Decoded Publish (Optional)

{"ts":12340,"src":"V","fn":"read","reg":"0x0003","count":6}
{"ts":12345,"src":"E","fn":"read_resp","reg":"0x0003",
 "values":{"0x0003":682,"0x0004":712,"0x0005":52}}

Values like 682 and 712 likely represent °F × 10.

Phase 1.5: Active Proxy

VTouch READ  → Bridge forwards → ERV responds → Bridge forwards to VTouch
VTouch WRITE → Bridge forwards → ERV ACK → Bridge forwards to VTouch
             → Bridge also publishes decoded command to MQTT

Phase 2: Full Proxy (Controller Takes Over)

VTouch READ  → Bridge answers from local erv/state cache
VTouch WRITE → Bridge publishes erv/command/request → Controller executes
Controller publishes erv/state → Bridge updates cache → VTouch stays synced
