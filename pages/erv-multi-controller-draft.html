Multi-Controller for a Single ERV

A draft plan to support multiple remotes (VTTOUCH panel + dedicated ESP32-S3 hardware controller) while keeping a single RS-485 master on the ERV bus.

Core Idea

The ERV RS-485 bus must be mastered by exactly one device at a time. We keep a single “controller node” (ESP32-S3 or a small Pi) on the RS-485 bus, then let all other devices talk to the controller over MQTT. This preserves the single-bus requirement while enabling multiple UIs and control surfaces.

Roles

  • Controller (RS-485 Master): ESP32-S3 or Pi with RS-485 adapter, owns the ERV bus.
  • Clients: VTTOUCH panel, web UI, or a dedicated ESP32-S3 button/display device.
  • Broker: MQTT acts as the shared state and command bus.

Why This Works

  • Single RS-485 master avoids firmware lockouts.
  • All commands are logged and replayable via MQTT.
  • Clients stay in sync by subscribing to the same state topics.

High-Level Architecture

[Client A: VTTOUCH Panel]      [Client B: ESP32-S3 Buttons]     [Client C: Web UI]
              │                            │                             │
              └─────────────── MQTT (commands + state + audit log) ───────┘
                                             │
                                   [Controller: ESP32-S3]
                                             │
                                      RS-485 to ERV

Command + State Topics (Draft)

  • erv/command/request — client publishes desired change (mode, speed, boost, etc.)
  • erv/command/ack — controller acknowledges command receipt + validation
  • erv/state — controller publishes authoritative ERV state
  • erv/audit — append-only log of every command + source client
  • erv/errors — controller reports bus faults or rejected commands

Client Requirements

  • Publish commands with a unique client_id.
  • Subscribe to erv/state for UI sync.
  • Listen for erv/command/ack to confirm actions.

Controller Responsibilities

  • Validate command payloads (range, mode safety).
  • Serialize commands to RS-485 (one at a time).
  • Publish authoritative state after every change.
  • Write every command to the audit topic.

Draft: Dedicated ESP32-S3 Remote

  • Hardware: ESP32-S3 + small OLED/LCD + 3-5 buttons (Mode, Speed, Boost, Power).
  • Publishes commands to erv/command/request.
  • Subscribes to erv/state to display live ERV values.
  • Shows last command source from erv/audit (optional).

Open Questions

  • Do we allow VTTOUCH to publish commands, or run it in read-only mirror mode?
  • Should commands require confirmation if they conflict with ERV rules?
  • Do we store audit logs in SQLite (recommended) or only MQTT?
