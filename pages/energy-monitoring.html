Project Overview

This is a fully local, open-source home intelligence system built around two goals: minimize heating and cooling costs while maintaining measurable comfort. It integrates every layer of the home's HVAC stack — ventilation, heat pump, indoor air quality, and whole-home power monitoring — into a single data pipeline with a unified web dashboard.

The system uses two Broan ERV units connected via RS-485 to ESP32 microcontrollers running ESPHome, feeding ventilation metrics into an MQTT broker. Aqara temperature and humidity sensors provide per-room indoor conditions. A Mitsubishi Hyper Heat mini-split is controlled via its CN105 serial port, bypassing cloud dependencies entirely. Emporia Vue monitors per-circuit power consumption as an independent ground-truth validator. Open-Meteo provides free, no-key 48-hour weather forecasts used for proactive scheduling — pre-cooling before a heat wave, pre-heating before a cold front.

Everything is logged to SQLite, exposed through a Python FastAPI backend, and viewable through a local web dashboard. No subscriptions, no cloud accounts required after initial setup.

System overview

Hardware & data flow

Ventilation

Broan B210E75RT ERV ×2

Each unit communicates over RS-485 Modbus via a MAX485 module wired to an ESP32-S3 running ESPHome. Exposes intake/exhaust temperature, CFM, power draw, and filter life. Commands (Min / Med / Max / Turbo) are sent over MQTT. The automation engine adjusts fan mode based on outdoor conditions and indoor air quality.

Heating & Cooling

Mitsubishi Hyper Heat

Controlled via the CN105 serial port using the SwiCago ESPHome HeatPump component — fully local, no cloud. Exposes mode, setpoint, true outdoor temperature from the compressor sensor, compressor frequency, and defrost state. The pre-heat scheduler raises setpoint before high-efficiency windows close.

Indoor Conditions

Aqara Hub V3 + Sensors

Per-room temperature and humidity sensors on Zigbee, bridged to MQTT via Home Assistant, the Aqara Cloud API, or Zigbee2MQTT. Sensors placed in the main floor, upstairs landing, basement, and near the ERV supply-air outlet give the automation engine room-level comfort data.

Power Validation

Emporia Vue

Per-circuit watt monitoring polled every 60 seconds via the pyemvue library. Completely independent of what the CN105 reports — this is the ground-truth validator. If the heat pump draws power when it shouldn't, the watchdog detects it from the wire and re-issues the off command.

Weather Forecasting

Open-Meteo — 48-hour forecast, free, no API key

Polled every 15 minutes. Returns current outdoor temp and humidity plus 48 hourly data points including temperature, dewpoint, precipitation probability, and wind speed. The ERV automation uses forecast to ventilate aggressively before a cold front hits and close down when outdoor humidity spikes. The heat pump scheduler uses it to identify high-COP heating windows up to 12 hours ahead — pre-loading thermal mass when efficiency is best, letting the house coast when COP is worst.

Stated success goals

What done looks like

1

Both ERV units report live data over MQTT

Intake and exhaust temperature, CFM, power draw, and filter life for each Broan unit are visible in the SQLite database and web dashboard with no gaps longer than 60 seconds. ERV fan mode can be commanded from the dashboard and confirmed via RS-485 response.

2

ERV automation runs weather-aware decisions 24/7

The 7-rule priority engine runs every 5 minutes. Every decision is logged with a human-readable reason. Manual overrides expire after 2 hours and automation resumes. The system can be paused from the dashboard without modifying configuration files.

3

Heat pump controllable and validated locally

CN105 adapter reports mode, setpoint, compressor frequency, and outdoor temperature to MQTT. Every commanded change is confirmed against both CN105 state and Emporia wattage within 90 seconds. Three failed confirmations trigger an alert. No Kumo Cloud or internet connection required for any control function.

4

Real COP curve built from actual operating data

After two weeks of logging, a SQL query groups Emporia wattage by 5°F outdoor temperature buckets to produce the unit's actual efficiency curve. This replaces spec-sheet estimates and gives the scheduler accurate cost-per-hour values at every outdoor temperature the system will encounter.

5

Pre-heat and pre-cool scheduling is forecast-driven

The scheduler detects when outdoor temperature will drop more than 20°F within 8 hours and pre-loads thermal mass while COP is high. Before heat waves it drops setpoint during the overnight cool window. Every scheduled change is logged with the forecast that triggered it, so the rationale is always traceable.

6

30-day cost trend is visible and downward

The dashboard shows actual $/day heating and cooling cost derived from Emporia circuit data and the local electricity rate. After 30 days of COP-aware scheduling, measured cost-per-heating-degree-day is lower than the pre-automation baseline captured during the data-collection phase.

7

Everything runs without internet (except weather)

MQTT, ESPHome, SQLite, FastAPI, and the web dashboard all operate on the local network with no cloud dependencies. The only outbound call is to api.open-meteo.com every 15 minutes. The system degrades gracefully: if the weather API is unreachable, the automation falls back to the most recent cached forecast.
