Phase 1A — Hardware Assembly
1

Set Buck Converter to 5V Before Touching the ESP32

MP1584 modules ship at an unknown output voltage. Connecting an unverified one to an ESP32 will destroy it instantly.
⚠️
Do this first, every time, for every modulePower the MP1584 from any 12V source (USB charger or the DROK set to 12V). Probe OUT+ and OUT− with a multimeter. Turn the brass pot screw slowly until you read 5.0–5.2V. Verify. Only then connect to the ESP32.
2

Wire MAX485 Module to ESP32-S3

Dupont female-to-female jumpers only. No soldering for this step.
Connection table
ESP32-S3MAX485Notes
5V→VCCRed wire
GND→GNDBlack wire
GPIO17 (TX)→DIGreen wire
GPIO18 (RX)→ROBlue wire
GPIO16→DE + RE (bridged)Purple — tie DE and RE together first
3

Connect to ERV Terminal Block

Green 6-position terminal block near the ERV control area. You use 4 of the 6 terminals.
🔌
VTTOUCHW touchscreen will go silentThe ERV RS-485 bus only responds to one controller. Once the ESP32 is active, the touchscreen stops responding — this is a hardware limitation in the ERV firmware. The web dashboard replaces it entirely. The OVR dry-contact terminals still work for physical override if needed.
🔄
A/B labeling is reversed from D+/D−On MAX485 modules: A = D− and B = D+. If you see alignment errors in ESPHome logs, swap these two wires first.
⚡
Connect 12V terminal lastWire everything else first. The ERV 12V terminal is the final connection once all other wiring is verified.

Phase 1C — MQTT Broker + Python Backend
5

Set Up Mosquitto MQTT Broker

Central message bus. All ESP32s publish here. Python backend subscribes here.
📦
Runs in a minimal LXCA Debian LXC with 256MB RAM is sufficient. No dedicated hardware required.
Install and configure
apt install mosquitto mosquitto-clients

# Create authenticated user:
mosquitto_passwd -c /etc/mosquitto/passwd erv_monitor

# /etc/mosquitto/conf.d/erv.conf
listener 1883
allow_anonymous false
password_file /etc/mosquitto/passwd

systemctl restart mosquitto && systemctl enable mosquitto

# Verify data is flowing from ESP32:
mosquitto_sub -h localhost -u erv_monitor -P yourpassword -t "erv/#" -v
# Should see: erv/unit1/sensor/intake_temperature 68.4

Phase 1D — Aqara Indoor Sensor Integration
A

Bridge Aqara Hub V3 Sensors to MQTT

You already have sensors placed throughout the house. This step pulls that data into the same MQTT pipeline as the ERV data so the automation engine can use room-by-room temperature and humidity.
🏠
Three paths — choose oneAll three end at the same place: your Aqara sensor readings on MQTT topics. Pick based on what you're comfortable running.
Path A — Home Assistant (recommended, most future-proof)

Install Home Assistant OS in its own LXC or VM. Add the official Aqara integration (Settings → Integrations → Aqara). Enable the MQTT integration in HA pointed at your Mosquitto broker. HA republishes every Aqara sensor automatically.

# HA MQTT topic pattern for Aqara sensors:
homeassistant/sensor/aqara_living_room_temp/state
homeassistant/sensor/aqara_bedroom_humidity/state
# Add these topic patterns to ERV_UNITS in erv_monitor.py
Path B — Aqara Cloud API direct poll (simplest, no HA needed)
# pip install python-aqara  (or use requests directly)
# Aqara developer.aqara.com — create app, get credentials

import httpx, asyncio

async def poll_aqara(mqtt_client):
    # POST to Aqara OpenAPI every 60 seconds
    headers = {"Authorization": f"Bearer {ACCESS_TOKEN}"}
    r = await client.post(
        "https://open-usa.aqara.com/v3.0/open/api",
        json={"intent": "query.resource.value",
              "data": [{"subjectId": DEVICE_ID,
                         "resourceIds": ["4.1.85", "0.1.85"]}]}
              # 4.1.85 = temperature, 0.1.85 = humidity
    )
    for sensor in r.json()["result"]:
        mqtt_client.publish(
            f"aqara/{sensor['subjectId']}/{sensor['resourceId']}",
            sensor["value"]
        )
    await asyncio.sleep(60)

Phase 1E — Open-Meteo Weather Integration
B

Open-Meteo Forecast Polling

Free, no API key, no rate limit concerns at 15-minute polling intervals. Returns current conditions and 48-hour hourly forecast — exactly what ERV decisions need.
🌤️
Why forecast matters more than current conditionsThe ERV takes time to have an effect on indoor conditions. A decision made on current outdoor temp alone is reactive. A decision made knowing outdoor temp will drop 25°F in 4 hours is proactive — you ventilate aggressively now and close down before the cold front hits.
Add to erv_monitor.py — weather polling service
import httpx

# Eastern PA coordinates — replace with yours
FORECAST_URL = (
    "https://api.open-meteo.com/v1/forecast"
    "?latitude=40.XX&longitude=-75.XX"
    "&hourly=temperature_2m,relativehumidity_2m,"
    "dewpoint_2m,precipitation_probability,windspeed_10m"
    "&forecast_days=2"
    "&temperature_unit=fahrenheit&windspeed_unit=mph"
    "&timezone=America/New_York"
)

async def fetch_forecast() -> dict:
    async with httpx.AsyncClient() as client:
        r = await client.get(FORECAST_URL, timeout=10)
        h = r.json()["hourly"]
        return {
            "outdoor_temp_f":    h["temperature_2m"][0],
            "outdoor_humidity":  h["relativehumidity_2m"][0],
            "dewpoint_f":        h["dewpoint_2m"][0],
            "precip_prob_now":   h["precipitation_probability"][0],
            "temp_in_2h":        h["temperature_2m"][2],
            "temp_in_4h":        h["temperature_2m"][4],
            "temp_in_6h":        h["temperature_2m"][6],
            "temp_in_12h":       h["temperature_2m"][12],
            "temp_in_24h":       h["temperature_2m"][24],
            "humidity_in_4h":    h["relativehumidity_2m"][4],
            "temp_min_12h":      min(h["temperature_2m"][:12]),
            "temp_max_12h":      max(h["temperature_2m"][:12]),
        }

async def weather_poll_loop():
    while True:
        weather = await fetch_forecast()
        # Store in db, publish to MQTT, feed into ERV decision engine
        await record_weather(weather)
        await erv_decision_engine(weather)
        await asyncio.sleep(900)  # poll every 15 minutes
Phase 1F — ERV Automation Rules
C

Weather-Aware ERV Decision Engine

Runs every 5 minutes. Combines outdoor conditions, indoor sensor data, and forecast to decide ERV fan mode. Always logs the reason for every decision.
PriorityConditionActionReason
1 — Safety outdoor_temp < 10°F Min Extreme cold — prevent heat loss and frosting
2 — Humidity indoor_rh > 55% Max Indoor humidity too high regardless of outdoor conditions
3 — Forecast temp_in_4h < outdoor_temp − 20°F Max → then Min Cold front incoming — ventilate while cheap, close down before it hits
4 — Moisture outdoor_rh > indoor_rh + 20% Min Outdoor air much more humid — bringing it in worsens indoor conditions
5 — Ideal 45°F < outdoor_temp < 75°F AND outdoor_rh < 65% Max Best possible outdoor air — ventilate aggressively, energy cost near zero
6 — Cold 10°F < outdoor_temp < 45°F Med Maintain air quality without excessive heat exchange load
7 — Default all other conditions Med Baseline ventilation
Decision engine implementation
async def erv_decision_engine(weather: dict):
    indoor_rh   = await get_avg_indoor_humidity()   # from Aqara
    outdoor_f   = weather["outdoor_temp_f"]
    outdoor_rh  = weather["outdoor_humidity"]
    temp_in_4h  = weather["temp_in_4h"]

    # Evaluate rules in priority order
    if outdoor_f < 10:
        mode, reason = "Min", "extreme cold"
    elif indoor_rh > 55:
        mode, reason = "Max", "high indoor humidity"
    elif temp_in_4h < outdoor_f - 20:
        mode, reason = "Max", "cold front in 4h — pre-ventilate"
        # Schedule a switch to Min in 3.5 hours
        asyncio.create_task(delayed_mode_set("Min", delay_minutes=210))
    elif outdoor_rh > indoor_rh + 20:
        mode, reason = "Min", "outdoor more humid than indoor"
    elif 45 < outdoor_f < 75 and outdoor_rh < 65:
        mode, reason = "Max", "ideal outdoor conditions"
    elif outdoor_f < 45:
        mode, reason = "Med", "cold — balanced ventilation"
    else:
        mode, reason = "Med", "default"

    for unit_id in ERV_UNITS:
        await set_erv_mode(unit_id, mode, reason)
